Design Principles

These principles guide every decision in the design system — from token structure to component API to documentation. When two options seem equally valid, these principles break the tie.

01

Consistency over speed

A component that's well-specified takes longer to build but prevents rework on both the design and code sides. Every component follows the same token hierarchy, the same naming convention, the same accessibility baseline. Shortcuts now become debt later.

Example
All 19 components use the same focus ring implementation: outline + outline-offset with 3 shared tokens. Change one token, every component updates.
Anti-pattern
Each component implements its own focus style — some use box-shadow, some use outline, some use border. Changing the focus ring means touching 19 files.
02

Accessible by default

WCAG 2.2 AA is the baseline, not an aspiration. Accessibility is built into tokens (contrast-validated color pairs), components (semantic HTML, ARIA attributes, keyboard support), and documentation (every component page has a WCAG checklist). There is no "accessibility phase" — it's part of every task.

Example
The Toggle component uses role="switch" and native checkbox behavior, giving screen readers correct semantics and keyboard users Space to toggle — no custom JS needed.
Anti-pattern
Building a toggle with <div onclick> and adding ARIA later "when we have time." The div has no keyboard support, no role, and no checked state.
03

Stack-agnostic

The DS outputs CSS custom properties and vanilla HTML. No framework dependency. Tokens are defined in JSON (DTCG format) and transformed to CSS via Style Dictionary. Components use .ds-* classes that work with any templating layer — React, Vue, Svelte, server-rendered HTML, or plain static pages.

Example
A button is <button class="ds-btn ds-btn--brand">. It works in a Next.js app, a WordPress theme, or a static HTML page. Zero JS required for styling.
Anti-pattern
Components that require import Button from '@ds/react' to render. Teams using Vue or plain HTML can't adopt the DS without a rewrite.
04

Tokens are the contract

Design tokens are the API between design and code. Figma uses variables that map 1:1 to CSS custom properties that map 1:1 to JSON source files. If a value doesn't have a token, it's either a gap in the system or an exception that needs documentation.

Example
A card's border radius is var(--ds-radius-lg) in CSS and bound to radius/lg in Figma. Change the token value in JSON, rebuild, and both sides update.
Anti-pattern
Figma uses 12px radius, CSS uses border-radius: 0.75rem hardcoded. They drift apart silently and nobody notices until a visual audit.
05

Decisions are recorded

Every architectural decision gets an ADR (Architecture Decision Record) in docs/decisions/. If there's no ADR, the decision wasn't made — it was improvised. ADRs capture context, alternatives considered, and consequences, so future contributors understand why, not just what.

Example
ADR-005 documents why the focus ring uses outline instead of box-shadow, what was considered, and what the impact is on Figma implementation.
Anti-pattern
Switching the focus ring approach in a commit message that says "fix focus ring" with no explanation of the reasoning or trade-offs.
06

Semantic over visual

Tokens and class names describe purpose, not appearance. --ds-feedback-error-default not --ds-red-600. ds-btn--danger not ds-btn--red. This allows the visual expression to change (a rebrand, a new palette) without breaking the API or the mental model.

Example
Using --ds-color-primary-default for the brand CTA. Swapping the brand file changes blue to cyan everywhere. No CSS edits needed.
Anti-pattern
Using --ds-color-blue-600 directly in a component. When the brand switches to cyan, every reference to blue needs manual replacement.
