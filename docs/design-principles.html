Design System

Design PrinciplesDesign Principles

Estes princípios guiam toda decisão no design system -- da estrutura de tokens à API de componentes à documentação. Quando duas opções parecem igualmente válidas, estes princípios desempatam.These principles guide every decision in the design system -- from token structure to component API to documentation. When two options seem equally valid, these principles break the tie.

01

Consistência acima de velocidadeConsistency over speed

Um componente bem especificado leva mais tempo para construir, mas previne retrabalho tanto no design quanto no código. Todo componente segue a mesma hierarquia de tokens, a mesma convenção de nomenclatura, o mesmo baseline de acessibilidade. Atalhos agora viram dívida depois.A component that's well-specified takes longer to build but prevents rework on both the design and code sides. Every component follows the same token hierarchy, the same naming convention, the same accessibility baseline. Shortcuts now become debt later.

ExemploExample
Todos os 19 componentes usam a mesma implementação de focus ring: outline + outline-offset com 3 tokens compartilhados. Mude um token, todo componente atualiza.All 19 components use the same focus ring implementation: outline + outline-offset with 3 shared tokens. Change one token, every component updates.
Anti-pattern
Cada componente implementa seu próprio estilo de foco -- alguns usam box-shadow, alguns usam outline, alguns usam border. Mudar o focus ring significa tocar em 19 arquivos.Each component implements its own focus style -- some use box-shadow, some use outline, some use border. Changing the focus ring means touching 19 files.
02

Acessível por padrãoAccessible by default

WCAG 2.2 AA é o baseline, não uma aspiração. Acessibilidade é construída nos tokens (pares de cores validados por contraste), componentes (HTML semântico, atributos ARIA, suporte a teclado) e documentação (toda página de componente tem um checklist WCAG). Não existe "fase de acessibilidade" -- faz parte de toda tarefa.WCAG 2.2 AA is the baseline, not an aspiration. Accessibility is built into tokens (contrast-validated color pairs), components (semantic HTML, ARIA attributes, keyboard support), and documentation (every component page has a WCAG checklist). There is no "accessibility phase" -- it's part of every task.

ExemploExample
O componente Toggle usa role="switch" e comportamento nativo de checkbox, dando a leitores de tela a semântica correta e aos usuários de teclado Space para alternar -- sem JS customizado.The Toggle component uses role="switch" and native checkbox behavior, giving screen readers correct semantics and keyboard users Space to toggle -- no custom JS needed.
Anti-pattern
Construir um toggle com <div onclick> e adicionar ARIA depois "quando tivermos tempo." A div não tem suporte a teclado, nenhum role e nenhum estado checked.Building a toggle with <div onclick> and adding ARIA later "when we have time." The div has no keyboard support, no role, and no checked state.
03

Agnóstico de stackStack-agnostic

O DS gera CSS custom properties e HTML vanilla. Sem dependência de framework. Tokens são definidos em JSON (formato DTCG) e transformados para CSS via Style Dictionary. Componentes usam classes .ds-* que funcionam com qualquer camada de template -- React, Vue, Svelte, HTML server-rendered ou páginas estáticas.The DS outputs CSS custom properties and vanilla HTML. No framework dependency. Tokens are defined in JSON (DTCG format) and transformed to CSS via Style Dictionary. Components use .ds-* classes that work with any templating layer -- React, Vue, Svelte, server-rendered HTML, or plain static pages.

ExemploExample
Um button é <button class="ds-btn ds-btn--brand">. Funciona em um app Next.js, um tema WordPress ou uma página HTML estática. Zero JS necessário para estilização.A button is <button class="ds-btn ds-btn--brand">. It works in a Next.js app, a WordPress theme, or a static HTML page. Zero JS required for styling.
Anti-pattern
Componentes que requerem import Button from '@ds/react' para renderizar. Times usando Vue ou HTML puro não conseguem adotar o DS sem reescrita.Components that require import Button from '@ds/react' to render. Teams using Vue or plain HTML can't adopt the DS without a rewrite.
04

Tokens são o contratoTokens are the contract

Design tokens são a API entre design e código. O Figma usa variáveis que mapeiam 1:1 para CSS custom properties que mapeiam 1:1 para arquivos JSON fonte. Se um valor não tem um token, é uma lacuna no sistema ou uma exceção que precisa de documentação.Design tokens are the API between design and code. Figma uses variables that map 1:1 to CSS custom properties that map 1:1 to JSON source files. If a value doesn't have a token, it's either a gap in the system or an exception that needs documentation.

ExemploExample
O border radius de um card é var(--ds-radius-12) no CSS e vinculado a radius/lg no Figma. Mude o valor do token no JSON, rebuild, e ambos os lados atualizam.A card's border radius is var(--ds-radius-12) in CSS and bound to radius/lg in Figma. Change the token value in JSON, rebuild, and both sides update.
Anti-pattern
O Figma usa 12px de radius, o CSS usa border-radius: 0.75rem hardcoded. Eles divergem silenciosamente e ninguém percebe até uma auditoria visual.Figma uses 12px radius, CSS uses border-radius: 0.75rem hardcoded. They drift apart silently and nobody notices until a visual audit.
05

Decisões são registradasDecisions are recorded

Toda decisão de arquitetura recebe um ADR (Architecture Decision Record) em docs/decisions/. Se não há ADR, a decisão não foi tomada -- foi improvisada. ADRs capturam contexto, alternativas consideradas e consequências, para que futuros contribuidores entendam o porquê, não apenas o quê.Every architectural decision gets an ADR (Architecture Decision Record) in docs/decisions/. If there's no ADR, the decision wasn't made -- it was improvised. ADRs capture context, alternatives considered, and consequences, so future contributors understand why, not just what.

ExemploExample
O ADR-005 documenta por que o focus ring usa outline ao invés de box-shadow, o que foi considerado e qual o impacto na implementação Figma.ADR-005 documents why the focus ring uses outline instead of box-shadow, what was considered, and what the impact is on Figma implementation.
Anti-pattern
Mudar a abordagem do focus ring em um commit message que diz "fix focus ring" sem nenhuma explicação do raciocínio ou trade-offs.Switching the focus ring approach in a commit message that says "fix focus ring" with no explanation of the reasoning or trade-offs.
06

Semântico acima de visualSemantic over visual

Tokens e nomes de classes descrevem propósito, não aparência. --ds-feedback-error-default não --ds-red-600. ds-btn--danger não ds-btn--red. Isso permite que a expressão visual mude (uma mudança de marca, uma nova paleta) sem quebrar a API ou o modelo mental.Tokens and class names describe purpose, not appearance. --ds-feedback-error-default not --ds-red-600. ds-btn--danger not ds-btn--red. This allows the visual expression to change (a rebrand, a new palette) without breaking the API or the mental model.

ExemploExample
Usar --ds-brand-default para o CTA da marca. Trocar o arquivo da marca muda de blue para cyan em todo lugar. Nenhuma edição CSS necessária.Using --ds-brand-default for the brand CTA. Swapping the brand file changes blue to cyan everywhere. No CSS edits needed.
Anti-pattern
Usar --ds-color-blue-600 diretamente em um componente. Quando a marca muda para cyan, toda referência a blue precisa de substituição manual.Using --ds-color-blue-600 directly in a component. When the brand switches to cyan, every reference to blue needs manual replacement.
