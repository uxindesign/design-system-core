Principios de designDesign Principles

Estes principios guiam toda decisao no design system -- da estrutura de tokens a API de componentes a documentacao. Quando duas opcoes parecem igualmente validas, estes principios desempatam.These principles guide every decision in the design system -- from token structure to component API to documentation. When two options seem equally valid, these principles break the tie.

01

Consistencia acima de velocidadeConsistency over speed

Um componente bem especificado leva mais tempo para construir, mas previne retrabalho tanto no design quanto no codigo. Todo componente segue a mesma hierarquia de tokens, a mesma convencao de nomenclatura, o mesmo baseline de acessibilidade. Atalhos agora viram divida depois.A component that's well-specified takes longer to build but prevents rework on both the design and code sides. Every component follows the same token hierarchy, the same naming convention, the same accessibility baseline. Shortcuts now become debt later.

ExemploExample
Todos os 19 componentes usam a mesma implementacao de focus ring: outline + outline-offset com 3 tokens compartilhados. Mude um token, todo componente atualiza.All 19 components use the same focus ring implementation: outline + outline-offset with 3 shared tokens. Change one token, every component updates.
Anti-pattern
Cada componente implementa seu proprio estilo de foco -- alguns usam box-shadow, alguns usam outline, alguns usam border. Mudar o focus ring significa tocar em 19 arquivos.Each component implements its own focus style -- some use box-shadow, some use outline, some use border. Changing the focus ring means touching 19 files.
02

Acessivel por padraoAccessible by default

WCAG 2.2 AA e o baseline, nao uma aspiracao. Acessibilidade e construida nos tokens (pares de cores validados por contraste), componentes (HTML semantico, atributos ARIA, suporte a teclado) e documentacao (toda pagina de componente tem um checklist WCAG). Nao existe "fase de acessibilidade" -- faz parte de toda tarefa.WCAG 2.2 AA is the baseline, not an aspiration. Accessibility is built into tokens (contrast-validated color pairs), components (semantic HTML, ARIA attributes, keyboard support), and documentation (every component page has a WCAG checklist). There is no "accessibility phase" -- it's part of every task.

ExemploExample
O componente Toggle usa role="switch" e comportamento nativo de checkbox, dando a leitores de tela a semantica correta e aos usuarios de teclado Space para alternar -- sem JS customizado.The Toggle component uses role="switch" and native checkbox behavior, giving screen readers correct semantics and keyboard users Space to toggle -- no custom JS needed.
Anti-pattern
Construir um toggle com <div onclick> e adicionar ARIA depois "quando tivermos tempo." A div nao tem suporte a teclado, nenhum role e nenhum estado checked.Building a toggle with <div onclick> and adding ARIA later "when we have time." The div has no keyboard support, no role, and no checked state.
03

Agnostico de stackStack-agnostic

O DS gera CSS custom properties e HTML vanilla. Sem dependencia de framework. Tokens sao definidos em JSON (formato DTCG) e transformados para CSS via Style Dictionary. Componentes usam classes .ds-* que funcionam com qualquer camada de template -- React, Vue, Svelte, HTML server-rendered ou paginas estaticas.The DS outputs CSS custom properties and vanilla HTML. No framework dependency. Tokens are defined in JSON (DTCG format) and transformed to CSS via Style Dictionary. Components use .ds-* classes that work with any templating layer -- React, Vue, Svelte, server-rendered HTML, or plain static pages.

ExemploExample
Um botao e <button class="ds-btn ds-btn--brand">. Funciona em um app Next.js, um tema WordPress ou uma pagina HTML estatica. Zero JS necessario para estilizacao.A button is <button class="ds-btn ds-btn--brand">. It works in a Next.js app, a WordPress theme, or a static HTML page. Zero JS required for styling.
Anti-pattern
Componentes que requerem import Button from '@ds/react' para renderizar. Times usando Vue ou HTML puro nao conseguem adotar o DS sem reescrita.Components that require import Button from '@ds/react' to render. Teams using Vue or plain HTML can't adopt the DS without a rewrite.
04

Tokens sao o contratoTokens are the contract

Design tokens sao a API entre design e codigo. O Figma usa variaveis que mapeiam 1:1 para CSS custom properties que mapeiam 1:1 para arquivos JSON fonte. Se um valor nao tem um token, e uma lacuna no sistema ou uma excecao que precisa de documentacao.Design tokens are the API between design and code. Figma uses variables that map 1:1 to CSS custom properties that map 1:1 to JSON source files. If a value doesn't have a token, it's either a gap in the system or an exception that needs documentation.

ExemploExample
O border radius de um card e var(--ds-radius-lg) no CSS e vinculado a radius/lg no Figma. Mude o valor do token no JSON, rebuild, e ambos os lados atualizam.A card's border radius is var(--ds-radius-lg) in CSS and bound to radius/lg in Figma. Change the token value in JSON, rebuild, and both sides update.
Anti-pattern
O Figma usa 12px de radius, o CSS usa border-radius: 0.75rem hardcoded. Eles divergem silenciosamente e ninguem percebe ate uma auditoria visual.Figma uses 12px radius, CSS uses border-radius: 0.75rem hardcoded. They drift apart silently and nobody notices until a visual audit.
05

Decisoes sao registradasDecisions are recorded

Toda decisao de arquitetura recebe um ADR (Architecture Decision Record) em docs/decisions/. Se nao ha ADR, a decisao nao foi tomada -- foi improvisada. ADRs capturam contexto, alternativas consideradas e consequencias, para que futuros contribuidores entendam o porque, nao apenas o que.Every architectural decision gets an ADR (Architecture Decision Record) in docs/decisions/. If there's no ADR, the decision wasn't made -- it was improvised. ADRs capture context, alternatives considered, and consequences, so future contributors understand why, not just what.

ExemploExample
O ADR-005 documenta porque o focus ring usa outline ao inves de box-shadow, o que foi considerado e qual o impacto na implementacao Figma.ADR-005 documents why the focus ring uses outline instead of box-shadow, what was considered, and what the impact is on Figma implementation.
Anti-pattern
Mudar a abordagem do focus ring em um commit message que diz "fix focus ring" sem nenhuma explicacao do raciocinio ou trade-offs.Switching the focus ring approach in a commit message that says "fix focus ring" with no explanation of the reasoning or trade-offs.
06

Semantico acima de visualSemantic over visual

Tokens e nomes de classes descrevem proposito, nao aparencia. --ds-feedback-error-default nao --ds-red-600. ds-btn--danger nao ds-btn--red. Isso permite que a expressao visual mude (uma mudanca de marca, uma nova paleta) sem quebrar a API ou o modelo mental.Tokens and class names describe purpose, not appearance. --ds-feedback-error-default not --ds-red-600. ds-btn--danger not ds-btn--red. This allows the visual expression to change (a rebrand, a new palette) without breaking the API or the mental model.

ExemploExample
Usar --ds-color-primary-default para o CTA da marca. Trocar o arquivo da marca muda de blue para cyan em todo lugar. Nenhuma edicao CSS necessaria.Using --ds-color-primary-default for the brand CTA. Swapping the brand file changes blue to cyan everywhere. No CSS edits needed.
Anti-pattern
Usar --ds-color-blue-600 diretamente em um componente. Quando a marca muda para cyan, toda referencia a blue precisa de substituicao manual.Using --ds-color-blue-600 directly in a component. When the brand switches to cyan, every reference to blue needs manual replacement.
