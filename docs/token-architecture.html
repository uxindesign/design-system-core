Token Architecture

Design tokens are organized in three layers. Each layer has a specific role, and tokens in one layer reference the layer below — never the other way around.

Three layers

Foundation
Raw values. The only layer with absolute values (hex, rem, px).

foundation.color.blue.600 → #2563EB
foundation.spacing.4 → 1rem
foundation.radius.md → 8px

~217 tokens across 10 categories.
→
Semantic
Intent and meaning. References foundation via aliases. Mode-aware (light/dark).

semantic.color.primary.default → {foundation.brand.primary}
semantic.text.default → {foundation.color.neutral.900} (light)
semantic.text.default → {foundation.color.neutral.50} (dark)

~104 tokens × 2 modes.
→
Component
Per-component tokens. References semantic, never foundation.

component.button.background.brand.default → {semantic.color.primary.default}
component.button.padding.md → {semantic.space.inset.md}

118 tokens across 11 component files.

How an alias chain works

When you press a Brand button, its background color resolves through this chain:

component button.background.brand.default
↓ references
semantic semantic.color.primary.default
↓ references
brand foundation.brand.primary
↓ references
foundation foundation.color.blue.600 → #2563EB

To rebrand, you swap brand.json. The entire chain updates: every component that uses primary.default gets the new color without touching any other file.

Brand sublayer

Brand sits inside Foundation as a bridge between palette and intent. It has only 2 tokens — no states, no modes.

{ "foundation": { "brand": { "primary": { "$value": "{foundation.color.blue.600}" }, "secondary": { "$value": "{foundation.color.purple.600}" } } } }

Only semantic.color.primary.default references foundation.brand.primary. All other states (hover, active, subtle) reference the foundation palette directly, because state shifts need specific palette steps that don't follow a formula.

Naming convention

Canonical pattern: {layer}.{category}.{type}.{variant}.{state}

The -default suffix rule
Every token ending in .default generates -default in CSS. No exceptions, no collapsing. semantic.color.primary.default becomes --ds-color-primary-default, not --ds-color-primary.

Light/dark mode

Semantic tokens have the same keys in both modes but different values. Foundation and component tokens are mode-invariant.

Mode-invariant tokens (same in both modes): all space.*, radius.*, border.width.*, focus.ring.width, focus.ring.offset.

Rules

#Rule
1Component tokens reference semantic, never foundation. Exception: hover/active states that need a specific palette step.
2Semantic tokens reference foundation, never hardcoded. No raw hex or rgba in the semantic layer.
3Foundation is the only layer with absolute values.
4Brand is part of foundation: 2 tokens (primary, secondary), no states, swappable per theme.
5Every token has $type per DTCG spec.
6Non-obvious tokens require $description.
7light.json and dark.json must have exactly the same key set.
8New categories or hierarchy breaks require an ADR.
