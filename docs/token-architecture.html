Token ArchitectureToken Architecture

Design tokens são organizados em três camadas: Foundation/Core, Semantic/System e Component. Cada camada tem um papel específico na cadeia de alias.Design tokens are organized in three layers: Foundation/Core, Semantic/System, and Component. Each layer has a specific role in the alias chain.

Camadas e cadeia de aliasLayers and alias chain

Três camadas, uma direção: Foundation/Core guarda valores brutos; Semantic/System traduz esses valores em decisões reutilizáveis; Component expõe o contrato anatômico consumido por CSS, Figma bindings e documentação de componente.Three layers, one direction: Foundation/Core holds raw values; Semantic/System turns them into reusable decisions; Component exposes the anatomical contract consumed by CSS, Figma bindings, and component documentation.

Structures e iconografiaStructures and iconography

Alguns namespaces da camada Component podem representar estruturas anatômicas compartilhadas, não componentes instanciáveis. Uma structure só existe quando há padrão nomeável, pelo menos três consumidores, paridade real de valores e documentação própria. O binding aponta direto para a structure; não se cria uma cadeia intermediária input/* → field/* quando o valor é realmente compartilhado.Some Component-layer namespaces may represent shared anatomical structures, not instantiable components. A structure only exists when there is a nameable pattern, at least three consumers, real value parity, and its own documentation. Bindings point directly to the structure; do not create an intermediate input/* → field/* chain when the value is truly shared.

Regras anti-regressão em FigmaFigma anti-regression rules
Componentes finais não devem usar glyph, fontes de ícone ou Icon Placeholder. Ícones vivos devem ser instâncias Lucide com width, height, cor e stroke-width bindados em tokens Component. Focus rings devem usar focus-ring/color/* e focus-ring/width; apenas o radius pode permanecer local quando acompanha o shape. A camada Component deve apontar somente para Semantic, nunca direto para Foundation ou para outro Component. Após exportar um snapshot Figma atualizado, rode npm run verify:figma-structure para validar essas invariantes.Final components must not use glyph, icon fonts, or Icon Placeholder. Live icons must be Lucide instances with width, height, color, and stroke-width bound to Component tokens. Focus rings must use focus-ring/color/* and focus-ring/width; only radius may stay local when it follows the shape. The Component layer must point only to Semantic, never directly to Foundation or another Component. After exporting an updated Figma snapshot, run npm run verify:figma-structure to validate these invariants.

Colisão entre token pai e subestadoParent token and substate collision
Quando o Figma expõe um token pai e um subestado no mesmo ramo, por exemplo border/error e border/error/hover, o repo pode manter a chave JSON compatível como border.error-hover enquanto o pipeline Style Dictionary não suportar token com $value e filhos de forma ponta a ponta. A CSS custom property permanece estável como --ds-border-error-hover. Esta é uma exceção técnica de serialização, não um padrão de naming para novos tokens.When Figma exposes a parent token and a substate in the same branch, for example border/error and border/error/hover, the repo may keep the compatible JSON key as border.error-hover until the Style Dictionary pipeline supports a token with $value and children end to end. The CSS custom property remains stable as --ds-border-error-hover. This is a technical serialization exception, not a naming pattern for new tokens.

RegrasRules

#RegraRule
1Consumidores finais consomem Component quando há contrato de componente. Exceções temporárias de migração ou bibliotecas internas precisam ser explícitas — nunca Foundation direto.Final consumers use Component when a component contract exists. Temporary migration exceptions or internal libraries must be explicit — never Foundation directly.
2Tokens semânticos referenciam foundation, nunca hardcoded. Nenhum hex ou rgba bruto na camada semântica (exceto rgba documentados em ADR).Semantic tokens reference foundation, never hardcoded. No raw hex or rgba in the semantic layer (except rgba documented in ADR).
3Foundation é a única camada com valores absolutos. Naming numérico (radius-8, spacing-16) — não t-shirt.Foundation is the only layer with absolute values. Numeric naming (radius-8, spacing-16) — not t-shirt.
4Semantic usa naming t-shirt (radius-md, space-sm) — distinto de Foundation pra evitar redundância.Semantic uses t-shirt naming (radius-md, space-sm) — distinct from Foundation to avoid redundancy.
5Todo token tem $type conforme spec DTCG.Every token has $type per DTCG spec.
6Tokens não óbvios requerem $description.Non-obvious tokens require $description.
7light.json e dark.json devem ter exatamente o mesmo conjunto de chaves.light.json and dark.json must have exactly the same key set.
8Novas categorias ou quebras de hierarquia requerem um ADR.New categories or hierarchy breaks require an ADR.
9Componente migrado não deve bindar cor Semantic diretamente. Crie um token Component com naming anatômico e alias para Semantic.A migrated component must not bind Semantic color directly. Create an anatomically named Component token and alias it to Semantic.
10Variáveis Figma nunca devem usar ALL_SCOPES; scopes devem ser explícitos e coerentes com a camada e a propriedade.Figma variables must never use ALL_SCOPES; scopes must be explicit and coherent with the layer and property.
11Structures vivem dentro da camada Component, não em collection separada. Só promova um namespace quando houver padrão anatômico nomeável, pelo menos três consumidores, paridade real de valores e documentação própria.Structures live inside the Component layer, not in a separate collection. Only promote a namespace when there is a nameable anatomical pattern, at least three consumers, real value parity, and its own documentation.
12icon não é structure: é vocabulário semântico para ícones. Componentes finais devem expor tokens próprios ou structures adequadas; biblioteca de ícones e Icon swap podem usar icon/* diretamente.icon is not a structure: it is semantic vocabulary for icons. Final components must expose their own tokens or appropriate structures; icon libraries and Icon swap may use icon/* directly.
13Antes de publicar ou espelhar um componente do Figma no repo, valide por evidência estrutural: todos os variants, bindings, sublayers, propriedades expostas, ícones e exceções reais devem ser auditados. Não considere o componente validado por amostragem ou por inferência do padrão majoritário.Before publishing or mirroring a Figma component into the repo, validate it through structural evidence: every variant, binding, sublayer, exposed property, icon, and real exception must be audited. Do not consider a component validated through sampling or by inferring from the majority pattern.
